# Session Log

Running dated log of work sessions on this codebase. Newest entries at the top.

## 2026-07-02

- Built the client portal: `src/app/client/layout.tsx` (branded shell + nav), `src/app/client/page.tsx` (home redirect), and `src/app/client/jobs/page.tsx` (read-only jobs view scoped to the signed-in client). Created a real test client login and confirmed the flow end-to-end: client logs in → sees only their own jobs in `/client/jobs`. First fully verified end-to-end client flow in the app.
- Added a shared `src/components/LogoutButton.tsx` and wired it into all three portal layouts (admin, cleaner, client).
- Fixed the admin logout button rendering as an invisible/blank box: it was missing `color: 'inherit'`, so the unstyled `<button>` fell back to native UA button text color instead of inheriting the page's text color like the `<Link>` nav items do.
- Added `src/proxy.ts` (Next.js 16 renamed `middleware.ts` → `proxy.ts`) for server-side, session-based route protection across `/admin`, `/cleaner`, and `/client` — unauthenticated requests are redirected to `/admin/login` before any page renders. Uses `supabase.auth.getUser()` (revalidates against Supabase Auth) rather than trusting a session cookie directly.
- Audited live RLS policies via `pg_policies` (per the new schema-verification rule below — did not trust the `0003_rls_phase2_policies.sql` migration file as current). Confirmed every SELECT policy requires `auth.uid()` to match an owner or the admin role, with no anonymous or cross-role access on any table except the intentionally public booking insert. The missing route protection above was therefore a UX gap, not a data leak.
- Added `CLAUDE.md` with standing operating rules for all future sessions: never trust `supabase/` migration files as the live schema source of truth, build-before-commit, restart-then-commit-then-push order, proactive security/UX flags, manual test suggestions, docs upkeep, and no secrets pasted into chat.
- Added role-based route gating to `src/proxy.ts`: beyond "is anyone logged in," it now looks up the caller's role in `user_roles` and redirects a logged-in user of one role away from another portal's routes to their own home (admin → `/admin/clients`, cleaner → `/cleaner/inbox`, client → `/client/jobs`). Factored the role→home mapping into a shared `src/lib/roleHome.ts` so `proxy.ts` and the login page can't drift apart.
- Wrote `supabase/0005_schema_catchup.sql`: idempotent migration documenting the live production schema (tables, columns, defaults, FKs, RLS enable + exact policy bodies) as ground truth, superseding the stale 0001-0003 files. Confirmed every column default and RLS policy clause against live `information_schema`/`pg_policies` output rather than assuming. Not yet run against any database — it's a no-op-on-live catch-up file, first real test would be against a fresh environment.
- Added login auto-redirect by role (`src/app/admin/login/page.tsx`) — signing in now sends the user straight to their portal home instead of leaving them on the login page; if no role resolves, the account is signed back out with an error rather than left in a stuck logged-in-nowhere-to-go state.
- Built an in-app forgot-password flow: a "Forgot password?" link on the login page (enumeration-safe — always shows the same confirmation message) and a new root-level `src/app/reset-password/page.tsx` (deliberately outside `/admin`, `/cleaner`, `/client` so `proxy.ts`'s role gating can't interrupt the recovery flow). Confirmed via `@supabase/ssr`'s source that PKCE-flow recovery links still emit the `PASSWORD_RECOVERY` auth event automatically, so no manual `exchangeCodeForSession` call was needed.
- Added Stripe test-mode credentials to `.env.local` and installed the `stripe` package. Along the way, discovered and fixed a pre-existing latent bug in `package.json`: `@supabase/supabase-js`, `@supabase/ssr`, and `@supabase/auth-helpers-nextjs` were each listed twice (harmless in practice since JSON parsers take the last duplicate key, but `npm install stripe` deduplicated them as a side effect).
- Built Stripe invoicing end-to-end, triggered manually by an admin after a job is marked `completed` (not pay-upfront-at-booking, per earlier project decision): `supabase/0006_stripe_invoicing.sql` adds `clients.stripe_customer_id`, `jobs.stripe_invoice_id`, and `jobs.payment_status` (CHECK-constrained to unpaid/invoiced/paid/failed); `src/app/api/stripe/send-invoice/route.ts` (admin-only, RLS-scoped via the existing cookie-session pattern, no service-role key) finds-or-creates the Stripe customer, creates a draft invoice, attaches the job as a line item, finalizes, and sends it via Stripe's own hosted email; `src/app/api/stripe/webhook/route.ts` verifies Stripe's signature and updates `payment_status` on `invoice.paid`/`invoice.payment_failed` — this one route uses the Supabase service-role key (only place in the app that does), since Stripe calls it server-to-server with no user session for RLS to key off. Admin and client Jobs pages both show a new Payment column. Webhook isn't testable yet — needs `STRIPE_WEBHOOK_SECRET` and `SUPABASE_SERVICE_ROLE_KEY` added and `stripe listen --forward-to` running locally, since there's still no domain/HTTPS for a real Stripe Dashboard webhook.
